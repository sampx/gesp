<decisions>
## Implementation Decisions

### D-R1: 修复 Sessions 表缺少 expires_at 索引
Sessions 表未定义 `expires_at` 索引，TTL 过期查询可能全表扫描。
- 添加 `CREATE INDEX idx_sessions_expires_at ON sessions(expires_at)` 用于 TTL 过期清理
- 决策来源：REVIEWS.md Plan 02 Concern [HIGH]

### D-R2: 明确 username UNIQUE 约束
Users 表 `username` 未显式标记 UNIQUE 约束（仅有注释说明）。
- 在 Drizzle schema 中为 username 字段添加 `.unique()` 约束
- 决策来源：REVIEWS.md Plan 02 Concern [MEDIUM]（升级处理）

### D-R3: Sessions 表 user_id 添加 ON DELETE CASCADE
Sessions 表 `user_id` 外键未定义 `ON DELETE CASCADE`，用户删除时会产生孤儿 session。
- 修改外键关系为 `.references(() => users.id, { onDelete: 'cascade' })`
- 决策来源：REVIEWS.md Plan 02 Concern [MEDIUM]（升级处理）

### D-R4: Session TTL 双重验证逻辑
Session middleware 未详细说明 TTL 双重验证逻辑。
- 请求时先解析 cookie → 查 DB 验证 session_id 存在且未过期 → 比较当前时间与 expires_at → 若有效，可选择刷新 TTL（滑动过期）
- Session ID 使用 `crypto.getRandomValues()` 生成 32 字节随机值，base64url 编码
- 决策来源：REVIEWS.md Plan 03 Concern [HIGH] x2

### D-R5: 注册接口用户名冲突处理
注册接口未考虑用户名冲突处理。
- UNIQUE 约束违反时返回 409 Conflict，错误消息模糊化
- 决策来源：REVIEWS.md Plan 04 Concern [HIGH]

### D-R6: 登录接口安全策略
登录接口未说明密码错误时的响应策略。
- 登录失败返回通用错误：`"用户名或密码错误"`，不泄露用户是否存在
- 决策来源：REVIEWS.md Plan 04 Concern [HIGH]

### D-R7: 默认密码安全策略
默认密码策略未定义，生产环境存在安全风险。
- 生产环境未设置 `ADMIN_PASSWORD` 时抛错拒绝启动，仅开发环境允许默认值
- 首次启动时若使用默认密码输出强烈警告
- 决策来源：REVIEWS.md Plan 05 Concern [HIGH]

### D-R8: 验证界面设计
设计一个 Web 页面供用户验证后续每个阶段的功能。
- 界面应包含 Phase 1 的认证功能验证（注册、登录、会话管理）
- 界面设计应可扩展，方便后续阶段添加新的验证项
- 技术栈：使用现有 Hono backend + 简单前端页面（不引入新框架）
- 界面放在 `/debug` 或 `/verify` 路径下，生产环境应禁用
- 决策来源：用户需求

## Specific Ideas

### 验证界面设计参考
- 页面路径：`/debug`（开发环境默认启用，生产环境通过环境变量 `ENABLE_DEBUG=true` 开启）
