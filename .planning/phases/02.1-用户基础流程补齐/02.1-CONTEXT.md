# Phase 2.1: 用户基础流程补齐 - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

补齐注册→登录→使用→登出的完整用户旅程，加上管理员对教员/学员账号的管理能力，使系统具备基本可用性。包含：学员/教员自助注册、登出功能、密码修改、管理员添加和管理用户账号。不包含数据权限控制和功能权限配置（未来阶段）。

</domain>

<decisions>
## Implementation Decisions

### 注册流程
- **D-01:** 独立 `/register` 页面，登录页底部加"没有账号？去注册"链接，注册页底部加"已有账号？去登录"链接
- **D-02:** 注册表单只含 username + password + 确认密码，display_name 自动设为 username（后续可在设置页修改），降低注册门槛
- **D-03:** 注册成功后自动登录 + 跳转对应首页（学员→`/student/dashboard`，教员→`/admin/dashboard`），复用 backend 已有的 register+session 逻辑
- **D-04:** 注册页角色选择只展示学员和教员（管理员不可自助注册），教员注册后跳转管理端

### 导航与登出
- **D-05:** 学员端顶部 Navbar：左侧 logo/系统名称，右侧用户头像（首字母）+ display_name + 下拉菜单（设置、登出）
- **D-06:** 管理端侧边栏底部固定区域：separator + 用户头像 + 用户名 + 角色标签（"超级管理员"或"教员"）+ 登出按钮，始终可见不随导航滚动
- **D-07:** 登出点击即执行，不需确认弹窗。调用 `/api/auth/logout` → 清除 cookie → 跳转 `/login`

### 密码修改
- **D-08:** 学员端 `/student/settings`，管理端 `/admin/settings`，各一个设置页面，第一个 section 就是"修改密码"
- **D-09:** 表单三字段：旧密码 + 新密码 + 确认新密码（REQUIREMENTS 要求验证旧密码）
- **D-10:** 修改成功后 toast 提示"密码修改成功"，保持当前登录状态，不跳转登录页

### 账号管理
- **D-11:** `/admin/students` 列表页，顶部统一"添加用户"按钮 → Dialog 内选择角色（学员/教员）→ 填写用户名+密码+确认密码
- **D-12:** 列表精简列：用户名、角色标签（学员/教员）、状态（启用/禁用 badge）、注册时间、操作列（禁用/启用 | 重置密码）
- **D-13:** 重置密码：点击后弹 Dialog，输入新密码 + 确认新密码，提交后 toast 提示
- **D-14:** `/admin/settings` 包含两个 section：修改密码（Phase 2.1 实现）+ 系统配置（Phase 7 占位）

### 角色架构
- **D-15:** 角色层级：ROOT(100) 超级管理员 = 系统内置，拥有全部权限；ADMIN(10) 教员 = 与 root 共用管理端界面，将来有数据权限和功能权限配置；STUDENT(1) 学员
- **D-16:** Phase 2.1 不实现数据/功能权限控制，但代码结构要为将来权限分层预留扩展点（如 API 区分角色、前端根据角色决定按钮可见性）
- **D-17:** root 可添加教员和学员，教员将来只能添加学员（Phase 2.1 暂不做权限限制，全部开放）
- **D-18:** Seed root 用户名从 `admin` 改为 `root`，环境变量从 `ADMIN_USERNAME/ADMIN_PASSWORD/ADMIN_DISPLAY_NAME` 改为 `ROOT_USERNAME/ROOT_PASSWORD/ROOT_DISPLAY_NAME`

### the agent's Discretion
- Navbar/sidebar 的 exact 样式和间距
- 表单验证错误提示的具体文案
- 空列表状态的展示方式
- Settings 页面的布局细节

</decisions>

## Existing Code Insights

### Reusable Assets
- `@/components/ui/*` — shadcn/ui 完整组件库（Card、Button、Input、Label、Dialog、Badge、Table、DropdownMenu 等），注册页和设置页可直接复用
- `@/components/role-card.tsx` — 角色选择卡组件（登录页已用），注册页可复用
- `loginAction` pattern — Server Action + fetch backend + cookie 设置 + redirect 模式，注册页可复用
- `AdminAuth()` / `StudentAuth()` — 角色守卫中间件，新增 API 路由直接使用
- `success()` / `error()` — 统一响应工具
- `hashPassword()` / `verifyPassword()` — 密码工具

### Established Patterns
- Server Actions 在 `actions.ts` 文件中定义，使用 `"use server"` 指令
- 页面组件在 `page.tsx`，layout 组件在 `layout.tsx`
- 表单使用 react-hook-form + Zod 验证（参考 `knowledge-form.tsx`）
- API 路由使用 Hono + zValidator + hono-openapi

### Integration Points
- `/register` 路由需加入 middleware.ts 的 PUBLIC_PATHS
- Navbar 需在 student/layout.tsx 中引入
- AdminSidebar 需修改，加底部用户信息区
- 新增 backend API：`POST /api/auth/change-password`、`POST /api/admin/users`（添加用户）、`PUT /api/admin/users/:id/status`（启用/禁用）、`POST /api/admin/users/:id/reset-password`（重置密码）

</code_context>

<specifics>
## Specific Ideas

- 角色体系澄清：ROOT(100) 超级管理员、ADMIN(10) 教员、STUDENT(1) 学员。教员与 root 共用管理端界面，将来有数据权限和功能权限分层
- Seed root 用户名必须从 `admin` 改为 `root`，避免与将来教员角色的 `admin` 概念冲突
- 环境变量改为 `ROOT_USERNAME`、`ROOT_PASSWORD`、`ROOT_DISPLAY_NAME`
- 注册页视觉风格延续登录页的角色卡设计，但只展示学员和教员
- 学员端 Navbar 下拉菜单用 shadcn DropdownMenu 组件

</specifics>
