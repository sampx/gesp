# Phase 02.1: Code Review Report

**Reviewed:** 2026-04-25T04:46:08Z
**Depth:** standard
**Files Reviewed:** 20
**Status:** issues_found

## Summary

Reviewed 20 source files covering user authentication flow: login/register pages (server actions + client components), admin/student settings, password management dialogs, backend auth routes/services, middleware, and tests. The backend code is well-structured with proper security practices: bcrypt hashing, httpOnly cookies, generic error messages preventing user enumeration, Zod validation on all endpoints, and role-based access control.

**Key Concern:** All client-side API calls (admin students page, add-user dialog, password change form, reset-password dialog) rely on reading `session_id` from `document.cookie` and passing it via manual Cookie header. The session cookie is set with `httpOnly: true`, making it invisible to `document.cookie` — these calls will receive `undefined` for sessionId and make unauthenticated requests. This breaks all client-side authenticated API calls unless there is infrastructure-level proxying (e.g., Next.js rewrites) that was not included in the reviewed files.

## Critical Issues

### CR-01: Client-side components cannot read httpOnly session cookie — all client-side authenticated API calls will fail

**Files:**
- `apps/web/src/app/admin/students/page.tsx:71-74`
- `apps/web/src/components/add-user-dialog.tsx:66-69`
- `apps/web/src/components/password-change-form.tsx:37-40`
- `apps/web/src/components/reset-password-dialog.tsx:54-57`

**Issue:** The login/register server actions set the session cookie with `httpOnly: true`:

```typescript
// login/actions.ts:37-42
cookieStore.set("session_id", sessionId, {
  path: "/",
  httpOnly: true,    // ← JavaScript cannot read this
  sameSite: "strict",
  maxAge: 7 * 24 * 60 * 60,
});
```

Four client components attempt to read this cookie via `document.cookie`:

```typescript
// admin/students/page.tsx:71-74
const sessionId = document.cookie
  .split(";")
  .find((c) => c.trim().startsWith("session_id="))
  ?.split("=")[1];   // Always undefined — httpOnly prevents JS access
```

When `sessionId` is `undefined`, the code falls through to:
```typescript
...(sessionId ? { Cookie: `session_id=${sessionId}` } : {}),  // Empty object — no auth
```

The fetch request goes to the backend without authentication, receives 401, and shows a generic error toast. All admin management features (user list, add user, password change, password reset) will silently fail.

**Additionally**, even if `document.cookie` could read the session_id, manually setting the `Cookie` header in browser `fetch()` is blocked by the browser — `Cookie` is a [forbidden header name](https://fetch.spec.whatwg.org/#forbidden-header-name) and is silently dropped.

**Fix:** Use Next.js server actions or Route Handlers as a proxy, keeping the httpOnly cookie on the frontend domain and forwarding requests server-to-backend:

```typescript
// apps/web/src/app/admin/users/actions.ts
"use server";
import { cookies } from "next/headers";

export async function getUsers(page = 1, pageSize = 20) {
  const cookieStore = await cookies();
  const sessionId = cookieStore.get("session_id")?.value;
  const backendUrl = process.env.BACKEND_URL || "http://localhost:3000";

  const res = await fetch(`${backendUrl}/api/admin/users?page=${page}&pageSize=${pageSize}`, {
    headers: { Cookie: `session_id=${sessionId}` },
  });
  return res.json();
}
```

Client components then call the server action instead of directly calling the backend.

## Warnings

### WR-01: Self-registration endpoint allows specifying role=ADMIN (10) without authentication

**File:** `packages/backend/src/routes/auth.ts:63-72`
**Issue:** The `/api/auth/register` endpoint accepts an optional `role` field in the request body. When provided, it calls `registerUserWithRole()` which allows both STUDENT(1) and ADMIN(10). There is no authentication required for this endpoint. A direct API call with `{"username": "hack", "password": "123456", "role": 10}` creates an ADMIN account.

The frontend (register/page.tsx) only shows "student" and "teacher" options, but the backend API has no way to distinguish between a legitimate frontend request and a direct API call.

**Fix:** Remove `role` from the public register endpoint, or require admin authentication for role elevation:

```typescript
// Option A: Remove role from public registration entirely
const registerBodySchema = z.object({
  username: z.string().min(3).max(50),
  password: z.string().min(6),
  display_name: z.string().min(1).max(100).optional(),
  // role removed — always STUDENT for self-registration
});

// Option B: Only allow STUDENT role in public register
if (role !== undefined && role !== ROLE.STUDENT) {
  return error(c, "Invalid role for self-registration", 400);
}
```

### WR-02: Admin operations lack ROOT user protection

**File:** `packages/backend/src/services/auth.service.ts:185-204, 206-229`
**Issue:** `toggleUserStatus()` and `resetUserPassword()` accept any user ID including the ROOT user (role=100). An ADMIN (role=10) could:
1. Disable the ROOT account via toggle status → locks out super admin
2. Reset the ROOT account's password → gains super admin access

The backend uses `AdminAuth()` middleware (role >= 10) but doesn't distinguish between target users' roles.

**Fix:**
```typescript
// In toggleUserStatus and resetUserPassword
if (user.role === ROLE.ROOT) {
  return { success: false, error: "Cannot modify ROOT user" };
}
```

### WR-03: Middleware redirects to /403 but no such page exists

**File:** `apps/web/src/middleware.ts:41`
**Issue:** When a non-admin user tries to access `/admin/*`, the middleware redirects to `/403`. No 403 page exists in the reviewed files, resulting in an infinite redirect loop (middleware catches `/403` → redirects to `/login` since it's not in PUBLIC_PATHS → but user is logged in → redirects... actually the cookie exists so it proceeds to `/403` which doesn't exist → Next.js shows default 404).

**Fix:** Either create a dedicated 403 page or redirect to a page that exists:
```typescript
// Option A: Create apps/web/src/app/403/page.tsx
// Option B: Redirect to login with error parameter
return NextResponse.redirect(new URL("/login?error=forbidden", request.url));
```
