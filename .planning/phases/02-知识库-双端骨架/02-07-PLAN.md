<task type="auto">
  <name>Fix LanceDB PUT update cache invalidation</name>
  <files>packages/backend/src/services/vector-store.ts</files>
  <action>
The root cause of the PUT 500 error: `LanceDBFileStore` is a singleton with a `tableCache` Map. The `update` method (line 225-232) does delete + re-insert using the cached table reference, but does NOT invalidate the cache afterward. Subsequent `getById` calls get the same cached table reference, which may return stale data or miss newly inserted records due to LanceDB's internal indexing behavior.

The fix: Invalidate the table cache after any mutation (insert, update, delete) to ensure subsequent queries open a fresh table reference.

**Changes to `packages/backend/src/services/vector-store.ts`:**

1. In the `update` method (line 225-232), add cache invalidation after the mutation:
```typescript
async update(tableName: string, id: string, data: Record<string, unknown>): Promise<void> {
    const safeId = sanitizeId(id);
    const table = await this.getOrConnectTable(tableName);
    await table.delete(`id = '${safeId}'`);
    await table.add([data]);
    // Invalidate cache — LanceDB table reference may return stale data
    // after delete+add mutation
    this.tableCache.delete(tableName);
}
```

2. In the `delete` method (line 234-238), add cache invalidation:
```typescript
async delete(tableName: string, id: string): Promise<void> {
    const safeId = sanitizeId(id);
    const table = await this.getOrConnectTable(tableName);
    await table.delete(`id = '${safeId}'`);
    // Invalidate cache after mutation
    this.tableCache.delete(tableName);
}
```

3. In the `insert` method (find it in the file), add cache invalidation after the `add` call. Look for the method that calls `table.add(records)` and add `this.tableCache.delete(tableName)` after it. This ensures the next read gets a fresh reference.

These three changes ensure that any read operation following a write will open a fresh table reference from LanceDB, avoiding stale cache issues.
